     946             : //
     947             : // Very often in Gecko, you'll find yourself in a situation where you want
     948             : // to invoke a method (with or without arguments) asynchronously.  You
     949             : // could write a small helper class inheriting from nsRunnable to handle
     950             : // all these details, or you could let NewRunnableMethod take care of all
     951             : // those details for you.
     952             : //
     953             : // The simplest use of NewRunnableMethod looks like:
     954             : //
     955             : //   nsCOMPtr<nsIRunnable> event =
     956             : //     mozilla::NewRunnableMethod("description", myObject, &MyClass::HandleEvent);
     957             : //   NS_DispatchToCurrentThread(event);
     958             : //
     959             : // Statically enforced constraints:
     960             : //  - myObject must be of (or implicitly convertible to) type MyClass
     961             : //  - MyClass must define AddRef and Release methods
     962             : //
     963             : // The "description" string should specify a human-readable name for the
     964             : // runnable; the provided string is used by various introspection tools
     965             : // in the browser.
     966             : //
     967             : // The created runnable will take a strong reference to `myObject`.  For
     968             : // non-refcounted objects, or refcounted objects with unusual refcounting
     969             : // requirements, and if and only if you are 110% certain that `myObject`
     970             : // will live long enough, you can use NewNonOwningRunnableMethod instead,
     971             : // which will, as its name implies, take a non-owning reference.  If you
     972             : // find yourself having to use this function, you should accompany your use
     973             : // with a proof comment describing why the runnable will not lead to
     974             : // use-after-frees.
     975             : //
     976             : // (If you find yourself writing contorted code to Release() an object
     977             : // asynchronously on a different thread, you should use the
     978             : // NS_ProxyRelease function.)
     979             : //
     980             : // Invoking a method with arguments takes a little more care.  The
     981             : // natural extension of the above:
     982             : //
     983             : //   nsCOMPtr<nsIRunnable> event =
     984             : //     mozilla::NewRunnableMethod("description", myObject, &MyClass::HandleEvent,
     985             : //                                arg1, arg2, ...);
     986             : //
     987             : // can lead to security hazards (e.g. passing in raw pointers to refcounted

     989             : // require you to specify the storage types used by the runnable, just as
     990             : // you would if you were writing out the class by hand:
     991             : //
     992             : //   nsCOMPtr<nsIRunnable> event =
