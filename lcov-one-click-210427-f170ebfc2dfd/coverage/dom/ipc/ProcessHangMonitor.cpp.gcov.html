      48             : /*
      49             :  * Basic architecture:
      50             :  *
      51             :  * Each process has its own ProcessHangMonitor singleton. This singleton exists
      52             :  * as long as there is at least one content process in the system. Each content
      53             :  * process has a HangMonitorChild and the chrome process has one
      54             :  * HangMonitorParent per process. Each process (including the chrome process)
      55             :  * runs a hang monitoring thread. The PHangMonitor actors are bound to this
      56             :  * thread so that they never block on the main thread.
      57             :  *
      58             :  * When the content process detects a hang, it posts a task to its hang thread,
      59             :  * which sends an IPC message to the hang thread in the parent. The parent
      60             :  * cancels any ongoing CPOW requests and then posts a runnable to the main
      61             :  * thread that notifies Firefox frontend code of the hang. The frontend code is
      62             :  * passed an nsIHangReport, which can be used to terminate the hang.
      63             :  *
      64             :  * If the user chooses to terminate a script, a task is posted to the chrome
      65             :  * process's hang monitoring thread, which sends an IPC message to the hang
      66             :  * thread in the content process. That thread sets a flag to indicate that JS
      67             :  * execution should be terminated the next time it hits the interrupt
      68             :  * callback. A similar scheme is used for debugging slow scripts. If a content
      69             :  * process or plug-in needs to be terminated, the chrome process does so
      70             :  * directly, without messaging the content process.
      71             :  */
