     308             : /**
     309             :  * The idle services goal is to notify subscribers when a certain time has
     310             :  * passed since the last user interaction with the system.
     311             :  *
     312             :  * On some platforms this is defined as the last time user events reached this
     313             :  * application, on other platforms it is a system wide thing - the preferred
     314             :  * implementation is to use the system idle time, rather than the application
     315             :  * idle time, as the things depending on the idle service are likely to use
     316             :  * significant resources (network, disk, memory, cpu, etc.).
     317             :  *
     318             :  * When the idle service needs to use the system wide idle timer, it typically
     319             :  * needs to poll the idle time value by the means of a timer.  It needs to
     320             :  * poll fast when it is in active idle mode (when it has a listener in the idle
     321             :  * mode) as it needs to detect if the user is active in other applications.
     322             :  *
     323             :  * When the service is waiting for the first listener to become idle, or when
     324             :  * it is only monitoring application idle time, it only needs to have the timer
     325             :  * expire at the time the next listener goes idle.
     326             :  *
     327             :  * The core state of the service is determined by:
     328             :  *
     329             :  * - A list of listeners.
     330             :  *
     331             :  * - A boolean that tells if any listeners are in idle mode.
     332             :  *
     333             :  * - A delta value that indicates when, measured from the last non-idle time,
     334             :  *   the next listener should switch to idle mode.
     335             :  *
     336             :  * - An absolute time of the last time idle mode was detected (this is used to
     337             :  *   judge if we have been out of idle mode since the last invocation of the
     338             :  *   service.
     339             :  *
     340             :  * There are four entry points into the system:
     341             :  *
     342             :  * - A new listener is registered.
     343             :  *
     344             :  * - An existing listener is deregistered.
     345             :  *
     346             :  * - User interaction is detected.
     347             :  *
     348             :  * - The timer expires.
     349             :  *
     350             :  * When a new listener is added its idle timeout, is compared with the next idle
     351             :  * timeout, and if lower, that time is stored as the new timeout, and the timer
     352             :  * is reconfigured to ensure a timeout around the time the new listener should
     353             :  * timeout.
     354             :  *
     355             :  * If the next idle time is above the idle time requested by the new listener
     356             :  * it won't be informed until the timer expires, this is to avoid recursive
     357             :  * behavior and to simplify the code.  In this case the timer will be set to
     358             :  * about 10 ms.
     359             :  *
     360             :  * When an existing listener is deregistered, it is just removed from the list
     361             :  * of active listeners, we don't stop the timer, we just let it expire.
     362             :  *
     363             :  * When user interaction is detected, either because it was directly detected or
     364             :  * because we polled the system timer and found it to be unexpected low, then we
     365             :  * check the flag that tells us if any listeners are in idle mode, if there are
     366             :  * they are removed from idle mode and told so, and we reset our state
     367             :  * caculating the next timeout and restart the timer if needed.
     368             :  *
     369             :  * ---- Build in logic
     370             :  *
     371             :  * In order to avoid restarting the timer endlessly, the timer function has
     372             :  * logic that will only restart the timer, if the requested timeout is before
     373             :  * the current timeout.
