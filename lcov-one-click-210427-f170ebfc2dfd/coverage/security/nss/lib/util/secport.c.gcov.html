     168             : /********************* Arena code follows *****************************
     169             :  * ArenaPools are like heaps.  The memory in them consists of large blocks,
     170             :  * called arenas, which are allocated from the/a system heap.  Inside an
     171             :  * ArenaPool, the arenas are organized as if they were in a stack.  Newly
     172             :  * allocated arenas are "pushed" on that stack.  When you attempt to
     173             :  * allocate memory from an ArenaPool, the code first looks to see if there
     174             :  * is enough unused space in the top arena on the stack to satisfy your
     175             :  * request, and if so, your request is satisfied from that arena.
     176             :  * Otherwise, a new arena is allocated (or taken from NSPR's list of freed
     177             :  * arenas) and pushed on to the stack.  The new arena is always big enough
     178             :  * to satisfy the request, and is also at least a minimum size that is
     179             :  * established at the time that the ArenaPool is created.
     180             :  *
     181             :  * The ArenaMark function returns the address of a marker in the arena at
     182             :  * the top of the arena stack.  It is the address of the place in the arena
     183             :  * on the top of the arena stack from which the next block of memory will
     184             :  * be allocated.  Each ArenaPool has its own separate stack, and hence
     185             :  * marks are only relevant to the ArenaPool from which they are gotten.
     186             :  * Marks may be nested.  That is, a thread can get a mark, and then get
     187             :  * another mark.
     188             :  *
     189             :  * It is intended that all the marks in an ArenaPool may only be owned by a
     190             :  * single thread.  In DEBUG builds, this is enforced.  In non-DEBUG builds,
     191             :  * it is not.  In DEBUG builds, when a thread gets a mark from an
     192             :  * ArenaPool, no other thread may acquire a mark in that ArenaPool while
     193             :  * that mark exists, that is, until that mark is unmarked or released.
     194             :  * Therefore, it is important that every mark be unmarked or released when
     195             :  * the creating thread has no further need for exclusive ownership of the
     196             :  * right to manage the ArenaPool.
     197             :  *
     198             :  * The ArenaUnmark function discards the ArenaMark at the address given,
     199             :  * and all marks nested inside that mark (that is, acquired from that same
     200             :  * ArenaPool while that mark existed).   It is an error for a thread other
     201             :  * than the mark's creator to try to unmark it.  When a thread has unmarked
     202             :  * all its marks from an ArenaPool, then another thread is able to set
     203             :  * marks in that ArenaPool.  ArenaUnmark does not deallocate (or "pop") any

     205             :  *
