      21             : /*
      22             :  * Gecko Profiler integration with the JS Engine
      23             :  * https://developer.mozilla.org/en/Performance/Profiling_with_the_Built-in_Profiler
      24             :  *
      25             :  * The Gecko Profiler (found in tools/profiler) is an implementation of a
      26             :  * profiler which has the ability to walk the C++ stack as well as use
      27             :  * instrumentation to gather information. When dealing with JS, however, the
      28             :  * profiler needs integration with the engine because otherwise it is very
      29             :  * difficult to figure out what javascript is executing.
      30             :  *
      31             :  * The current method of integration with the profiler is a form of
      32             :  * instrumentation: every time a JS function is entered, a bit of information
      33             :  * is pushed onto a stack that the profiler owns and maintains. This
      34             :  * information is then popped at the end of the JS function. The profiler
      35             :  * informs the JS engine of this stack at runtime, and it can by turned on/off
      36             :  * dynamically.
      37             :  *
      38             :  * The profiler stack has three parameters: a base pointer, a size, and a
      39             :  * maximum size. The stack is the ProfileEntry stack which will have
      40             :  * information written to it. The size location is a pointer to an integer
      41             :  * which represents the current size of the stack (number of valid frames).
      42             :  * This size will be modified when JS functions are called. The maximum
      43             :  * specified is the maximum capacity of the ProfileEntry stack.
      44             :  *
      45             :  * Throughout execution, the size of the stack recorded in memory may exceed the
      46             :  * maximum. The JS engine will not write any information past the maximum limit,
      47             :  * but it will still maintain the size of the stack. Profiler code is aware of
      48             :  * this and iterates the stack accordingly.
      49             :  *
      50             :  * There is some information pushed on the profiler stack for every JS function
      51             :  * that is entered. First is a char* pointer of a description of what function
      52             :  * was entered. Currently this string is of the form "function (file:line)" if
      53             :  * there's a function name, or just "file:line" if there's no function name
      54             :  * available. The other bit of information is the relevant C++ (native) stack
      55             :  * pointer. This stack pointer is what enables the interleaving of the C++ and
      56             :  * the JS stack. Finally, throughout execution of the function, some extra
      57             :  * information may be updated on the ProfileEntry structure.
      58             :  *
      59             :  * = Profile Strings
      60             :  *
      61             :  * The profile strings' allocations and deallocation must be carefully
      62             :  * maintained, and ideally at a very low overhead cost. For this reason, the JS
      63             :  * engine maintains a mapping of all known profile strings. These strings are
      64             :  * keyed in lookup by a JSScript*, but are serialized with a JSFunction*,
      65             :  * JSScript* pair. A JSScript will destroy its corresponding profile string when
      66             :  * the script is finalized.
      67             :  *
      68             :  * For this reason, a char* pointer pushed on the profiler stack is valid only
      69             :  * while it is on the profiler stack. The profiler uses sampling to read off
      70             :  * information from this instrumented stack, and it therefore copies the string
      71             :  * byte for byte when a JS function is encountered during sampling.
      72             :  *
      73             :  * = Native Stack Pointer
      74             :  *
      75             :  * The actual value pushed as the native pointer is nullptr for most JS
      76             :  * functions. The reason for this is that there's actually very little
      77             :  * correlation between the JS stack and the C++ stack because many JS functions
      78             :  * all run in the same C++ frame, or can even go backwards in C++ when going
      79             :  * from the JIT back to the interpreter.
      80             :  *
      81             :  * To alleviate this problem, all JS functions push nullptr as their "native
      82             :  * stack pointer" to indicate that it's a JS function call. The function
      83             :  * RunScript(), however, pushes an actual C++ stack pointer onto the profiler
      84             :  * stack. This way when interleaving C++ and JS, if the Gecko Profiler sees a
      85             :  * nullptr native stack pointer on the profiler stack, it looks backwards for
      86             :  * the first non-nullptr pointer and uses that for all subsequent nullptr
      87             :  * native stack pointers.
      88             :  *
      89             :  * = Line Numbers
      90             :  *
      91             :  * One goal of sampling is to get both a backtrace of the JS stack, but also
      92             :  * know where within each function on the stack execution currently is. For
      93             :  * this, each ProfileEntry has a 'pc' field to tell where its execution
      94             :  * currently is. This field is updated whenever a call is made to another JS
      95             :  * function, and for the JIT it is also updated whenever the JIT is left.
      96             :  *
      97             :  * This field is in a union with a uint32_t 'line' so that C++ can make use of
      98             :  * the field as well. It was observed that tracking 'line' via PCToLineNumber in
      99             :  * JS was far too expensive, so that is why the pc instead of the translated
     100             :  * line number is stored.
     101             :  *
     102             :  * As an invariant, if the pc is nullptr, then the JIT is currently executing
     103             :  * generated code. Otherwise execution is in another JS function or in C++. With
     104             :  * this in place, only the top entry of the stack can ever have nullptr as its
     105             :  * pc. Additionally with this invariant, it is possible to maintain mappings of
     106             :  * JIT code to pc which can be accessed safely because they will only be
     107             :  * accessed from a signal handler when the JIT code is executing.
     108             :  */
