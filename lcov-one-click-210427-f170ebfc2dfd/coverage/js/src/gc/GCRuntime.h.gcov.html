     230             : /*
     231             :  * GC Scheduling Overview
     232             :  * ======================
     233             :  *
     234             :  * Scheduling GC's in SpiderMonkey/Firefox is tremendously complicated because
     235             :  * of the large number of subtle, cross-cutting, and widely dispersed factors
     236             :  * that must be taken into account. A summary of some of the more important
     237             :  * factors follows.
     238             :  *
     239             :  * Cost factors:
     240             :  *
     241             :  *   * GC too soon and we'll revisit an object graph almost identical to the
     242             :  *     one we just visited; since we are unlikely to find new garbage, the
     243             :  *     traversal will be largely overhead. We rely heavily on external factors
     244             :  *     to signal us that we are likely to find lots of garbage: e.g. "a tab
     245             :  *     just got closed".
     246             :  *
     247             :  *   * GC too late and we'll run out of memory to allocate (e.g. Out-Of-Memory,
     248             :  *     hereafter simply abbreviated to OOM). If this happens inside
     249             :  *     SpiderMonkey we may be able to recover, but most embedder allocations
     250             :  *     will simply crash on OOM, even if the GC has plenty of free memory it
     251             :  *     could surrender.
     252             :  *
     253             :  *   * Memory fragmentation: if we fill the process with GC allocations, a
     254             :  *     request for a large block of contiguous memory may fail because no
     255             :  *     contiguous block is free, despite having enough memory available to
     256             :  *     service the request.
     257             :  *
     258             :  *   * Management overhead: if our GC heap becomes large, we create extra
     259             :  *     overhead when managing the GC's structures, even if the allocations are
     260             :  *     mostly unused.
     261             :  *
     262             :  * Heap Management Factors:
     263             :  *
     264             :  *   * GC memory: The GC has its own allocator that it uses to make fixed size
     265             :  *     allocations for GC managed things. In cases where the GC thing requires
     266             :  *     larger or variable sized memory to implement itself, it is responsible
     267             :  *     for using the system heap.
     268             :  *
     269             :  *   * C Heap Memory: Rather than allowing for large or variable allocations,
     270             :  *     the SpiderMonkey GC allows GC things to hold pointers to C heap memory.
     271             :  *     It is the responsibility of the thing to free this memory with a custom
     272             :  *     finalizer (with the sole exception of NativeObject, which knows about
     273             :  *     slots and elements for performance reasons). C heap memory has different
     274             :  *     performance and overhead tradeoffs than GC internal memory, which need
     275             :  *     to be considered with scheduling a GC.
     276             :  *
     277             :  * Application Factors:
     278             :  *
     279             :  *   * Most applications allocate heavily at startup, then enter a processing
     280             :  *     stage where memory utilization remains roughly fixed with a slower
     281             :  *     allocation rate. This is not always the case, however, so while we may
     282             :  *     optimize for this pattern, we must be able to handle arbitrary
     283             :  *     allocation patterns.
     284             :  *
     285             :  * Other factors:
     286             :  *
     287             :  *   * Other memory: This is memory allocated outside the purview of the GC.
     288             :  *     Data mapped by the system for code libraries, data allocated by those
     289             :  *     libraries, data in the JSRuntime that is used to manage the engine,
     290             :  *     memory used by the embedding that is not attached to a GC thing, memory
     291             :  *     used by unrelated processes running on the hardware that use space we
     292             :  *     could otherwise use for allocation, etc. While we don't have to manage
     293             :  *     it, we do have to take it into account when scheduling since it affects
     294             :  *     when we will OOM.
     295             :  *
     296             :  *   * Physical Reality: All real machines have limits on the number of bits
     297             :  *     that they are physically able to store. While modern operating systems
     298             :  *     can generally make additional space available with swapping, at some
     299             :  *     point there are simply no more bits to allocate. There is also the
     300             :  *     factor of address space limitations, particularly on 32bit machines.
     301             :  *
     302             :  *   * Platform Factors: Each OS makes use of wildly different memory
     303             :  *     management techniques. These differences result in different performance
     304             :  *     tradeoffs, different fragmentation patterns, and different hard limits
     305             :  *     on the amount of physical and/or virtual memory that we can use before
     306             :  *     OOMing.
     307             :  *
     308             :  *
     309             :  * Reasons for scheduling GC
     310             :  * -------------------------
     311             :  *
     312             :  *  While code generally takes the above factors into account in only an ad-hoc
     313             :  *  fashion, the API forces the user to pick a "reason" for the GC. We have a
     314             :  *  bunch of JS::gcreason reasons in GCAPI.h. These fall into a few categories
     315             :  *  that generally coincide with one or more of the above factors.
     316             :  *
     317             :  *  Embedding reasons:
     318             :  *
     319             :  *   1) Do a GC now because the embedding knows something useful about the
     320             :  *      zone's memory retention state. These are gcreasons like LOAD_END,
     321             :  *      PAGE_HIDE, SET_NEW_DOCUMENT, DOM_UTILS. Mostly, Gecko uses these to
     322             :  *      indicate that a significant fraction of the scheduled zone's memory is
     323             :  *      probably reclaimable.
     324             :  *
     325             :  *   2) Do some known amount of GC work now because the embedding knows now is
     326             :  *      a good time to do a long, unblockable operation of a known duration.
     327             :  *      These are INTER_SLICE_GC and REFRESH_FRAME.
     328             :  *
     329             :  *  Correctness reasons:
     330             :  *
     331             :  *   3) Do a GC now because correctness depends on some GC property. For
     332             :  *      example, CC_WAITING is where the embedding requires the mark bits
     333             :  *      to be set correct. Also, EVICT_NURSERY where we need to work on the tenured
     334             :  *      heap.
     335             :  *
     336             :  *   4) Do a GC because we are shutting down: e.g. SHUTDOWN_CC or DESTROY_*.
     337             :  *
     338             :  *   5) Do a GC because a compartment was accessed between GC slices when we
     339             :  *      would have otherwise discarded it. We have to do a second GC to clean
     340             :  *      it up: e.g. COMPARTMENT_REVIVED.
     341             :  *
     342             :  *  Emergency Reasons:
     343             :  *
     344             :  *   6) Do an all-zones, non-incremental GC now because the embedding knows it
     345             :  *      cannot wait: e.g. MEM_PRESSURE.
     346             :  *
     347             :  *   7) OOM when fetching a new Chunk results in a LAST_DITCH GC.
     348             :  *
     349             :  *  Heap Size Limitation Reasons:
     350             :  *
     351             :  *   8) Do an incremental, zonal GC with reason MAYBEGC when we discover that
     352             :  *      the gc's allocated size is approaching the current trigger. This is
     353             :  *      called MAYBEGC because we make this check in the MaybeGC function.
     354             :  *      MaybeGC gets called at the top of the main event loop. Normally, it is
     355             :  *      expected that this callback will keep the heap size limited. It is
     356             :  *      relatively inexpensive, because it is invoked with no JS running and
     357             :  *      thus few stack roots to scan. For this reason, the GC's "trigger" bytes
     358             :  *      is less than the GC's "max" bytes as used by the trigger below.
     359             :  *
     360             :  *   9) Do an incremental, zonal GC with reason MAYBEGC when we go to allocate
     361             :  *      a new GC thing and find that the GC heap size has grown beyond the
     362             :  *      configured maximum (JSGC_MAX_BYTES). We trigger this GC by returning
     363             :  *      nullptr and then calling maybeGC at the top level of the allocator.
     364             :  *      This is then guaranteed to fail the "size greater than trigger" check
     365             :  *      above, since trigger is always less than max. After performing the GC,
     366             :  *      the allocator unconditionally returns nullptr to force an OOM exception
     367             :  *      is raised by the script.
     368             :  *
     369             :  *      Note that this differs from a LAST_DITCH GC where we actually run out
     370             :  *      of memory (i.e., a call to a system allocator fails) when trying to
     371             :  *      allocate. Unlike above, LAST_DITCH GC only happens when we are really
     372             :  *      out of memory, not just when we cross an arbitrary trigger; despite
     373             :  *      this, it may still return an allocation at the end and allow the script
     374             :  *      to continue, if the LAST_DITCH GC was able to free up enough memory.
     375             :  *
     376             :  *  10) Do a GC under reason ALLOC_TRIGGER when we are over the GC heap trigger
     377             :  *      limit, but in the allocator rather than in a random call to maybeGC.
     378             :  *      This occurs if we allocate too much before returning to the event loop
     379             :  *      and calling maybeGC; this is extremely common in benchmarks and
     380             :  *      long-running Worker computations. Note that this uses a wildly
     381             :  *      different mechanism from the above in that it sets the interrupt flag
     382             :  *      and does the GC at the next loop head, before the next alloc, or
     383             :  *      maybeGC. The reason for this is that this check is made after the
     384             :  *      allocation and we cannot GC with an uninitialized thing in the heap.
     385             :  *
     386             :  *  11) Do an incremental, zonal GC with reason TOO_MUCH_MALLOC when we have
     387             :  *      malloced more than JSGC_MAX_MALLOC_BYTES in a zone since the last GC.
     388             :  *
     389             :  *
     390             :  * Size Limitation Triggers Explanation
     391             :  * ------------------------------------
     392             :  *
     393             :  *  The GC internally is entirely unaware of the context of the execution of
     394             :  *  the mutator. It sees only:
     395             :  *
     396             :  *   A) Allocated size: this is the amount of memory currently requested by the
     397             :  *      mutator. This quantity is monotonically increasing: i.e. the allocation
     398             :  *      rate is always >= 0. It is also easy for the system to track.
     399             :  *
     400             :  *   B) Retained size: this is the amount of memory that the mutator can
     401             :  *      currently reach. Said another way, it is the size of the heap
     402             :  *      immediately after a GC (modulo background sweeping). This size is very
     403             :  *      costly to know exactly and also extremely hard to estimate with any
     404             :  *      fidelity.
     405             :  *
     406             :  *   For reference, a common allocated vs. retained graph might look like:
     407             :  *
     408             :  *       |                                  **         **
     409             :  *       |                       **       ** *       **
     410             :  *       |                     ** *     **   *     **
     411             :  *       |           *       **   *   **     *   **
     412             :  *       |          **     **     * **       * **
     413             :  *      s|         * *   **       ** +  +    **
     414             :  *      i|        *  *  *      +  +       +  +     +
     415             :  *      z|       *   * * +  +                   +     +  +
     416             :  *      e|      *    **+
     417             :  *       |     *     +
     418             :  *       |    *    +
     419             :  *       |   *   +
     420             :  *       |  *  +
     421             :  *       | * +
     422             :  *       |*+
     423             :  *       +--------------------------------------------------
     424             :  *                               time
     425             :  *                                           *** = allocated
     426             :  *                                           +++ = retained
     427             :  *
     428             :  *           Note that this is a bit of a simplification
     429             :  *           because in reality we track malloc and GC heap
     430             :  *           sizes separately and have a different level of
     431             :  *           granularity and accuracy on each heap.
     432             :  *
     433             :  *   This presents some obvious implications for Mark-and-Sweep collectors.
     434             :  *   Namely:
     435             :  *       -> t[marking] ~= size[retained]
     436             :  *       -> t[sweeping] ~= size[allocated] - size[retained]
     437             :  *
     438             :  *   In a non-incremental collector, maintaining low latency and high
     439             :  *   responsiveness requires that total GC times be as low as possible. Thus,
     440             :  *   in order to stay responsive when we did not have a fully incremental
     441             :  *   collector, our GC triggers were focused on minimizing collection time.
     442             :  *   Furthermore, since size[retained] is not under control of the GC, all the
     443             :  *   GC could do to control collection times was reduce sweep times by
     444             :  *   minimizing size[allocated], per the equation above.
     445             :  *
     446             :  *   The result of the above is GC triggers that focus on size[allocated] to
     447             :  *   the exclusion of other important factors and default heuristics that are
     448             :  *   not optimal for a fully incremental collector. On the other hand, this is
     449             :  *   not all bad: minimizing size[allocated] also minimizes the chance of OOM
     450             :  *   and sweeping remains one of the hardest areas to further incrementalize.
     451             :  *
     452             :  *      EAGER_ALLOC_TRIGGER
     453             :  *      -------------------
     454             :  *      Occurs when we return to the event loop and find our heap is getting
     455             :  *      largish, but before t[marking] OR t[sweeping] is too large for a
     456             :  *      responsive non-incremental GC. This is intended to be the common case
     457             :  *      in normal web applications: e.g. we just finished an event handler and
     458             :  *      the few objects we allocated when computing the new whatzitz have
     459             :  *      pushed us slightly over the limit. After this GC we rescale the new
     460             :  *      EAGER_ALLOC_TRIGGER trigger to 150% of size[retained] so that our
     461             :  *      non-incremental GC times will always be proportional to this size
     462             :  *      rather than being dominated by sweeping.
     463             :  *
     464             :  *      As a concession to mutators that allocate heavily during their startup
     465             :  *      phase, we have a highFrequencyGCMode that ups the growth rate to 300%
     466             :  *      of the current size[retained] so that we'll do fewer longer GCs at the
     467             :  *      end of the mutator startup rather than more, smaller GCs.
     468             :  *
     469             :  *          Assumptions:
     470             :  *            -> Responsiveness is proportional to t[marking] + t[sweeping].
     471             :  *            -> size[retained] is proportional only to GC allocations.
     472             :  *
     473             :  *      ALLOC_TRIGGER (non-incremental)
     474             :  *      -------------------------------
     475             :  *      If we do not return to the event loop before getting all the way to our
     476             :  *      gc trigger bytes then MAYBEGC will never fire. To avoid OOMing, we
     477             :  *      succeed the current allocation and set the script interrupt so that we
     478             :  *      will (hopefully) do a GC before we overflow our max and have to raise
     479             :  *      an OOM exception for the script.
     480             :  *
